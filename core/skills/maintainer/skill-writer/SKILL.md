# Write the skills

You are the injector's **capability stage**. Take the skill requirements + script details from `task-analyser` and make the **smallest correct change** to the learned skills. The default is *not* "write a new skill" — it's "find what already covers this and reuse it."

**Compose with `skill-creator`, don't reinvent authoring.** For the mechanical work — scaffolding a `SKILL.md`, structuring its steps, and tuning its triggering `description` (and, for a substantial new skill, running its eval loop) — use the generic **`skill-creator`** skill (named here so the model loads it). `skill-writer` supplies the judgment `skill-creator` can't: *which* change to make per Rule A4, *which* existing skills to reuse, and the hebb conventions below (top-level `skills/` placement, bundled python that passes the gate, the `$CODE_BASE` env contract). You extend `skill-creator` with hebb's discipline; it does the boilerplate.

## Steps
1. **Survey what exists.** You already see every skill's `name` + `description` (Claude Code loads them). For each requirement, also read the candidates in full: learned skills under `skills/` and core skills under `core/skills/` (for reuse/composition targets — a learned skill *calls* a core skill rather than editing it to work; fixing the engine itself is a separate task, see `core/agents/hebb_injector.md` → *Fixing issues at the source*).
2. **Pick the A4 branch** per requirement:
   - **No coverage** → create a new skill.
   - **Exists but was never picked up** → *discovery* problem: fix the existing skill's **description** (and/or `paths`) so it surfaces — **not** its script.
   - **Picked up but fell short** → *capability* problem: fix its **script or steps**.
   - **Covered by composition** — branch on whether the chain carries judgment *between* its steps (`task-analyser` flags which):
     - **No judgment between steps** (A→B is a deterministic hand-off; B's inputs are a pure function of A's outputs) → build **one combined skill backed by a single bundled script** that runs the whole pipeline in one invocation. Extract each reusable stage into `learned/hebb_utils/<domain>/` (step 3) and import it from the combined script **and** from each constituent skill's entry script, so all share one implementation. **Build it on the chain's first occurrence — no recurrence required** (Rule A2/A4); the absence of intervening judgment is the trigger.
     - **Judgment between steps** (the agent inspects A's result to decide what B does) → write a **thin composite skill** that **names and invokes** the constituents in order so the model loads them — and only when the chain recurs or its glue causes friction. Promote to an agent only if Rule A1 holds and the role recurs (rare). **Oncall exception:** an oncall ticket-type flow gets its thin composite (a high-level per-type runbook skill, naming the flow + linking the KB page) on its **first** occurrence — a standing per-type rule (see `task-analyser`), so every oncall ticket type exposes one high-level entry skill regardless of recurrence.
     - Composition is **additive** either way: keep every constituent skill intact and independently usable — never delete or break a bottom skill, since other tasks still call the individual skills.
3. **Reuse before duplicating.** Decompose each requirement into the capabilities it needs and map each to an existing skill. *Example:* a "specific-table-search" need = **table-schema knowledge** + **running a query**. If a query-running skill already exists, **reuse it** (name it in the new skill's body) and add only the schema-specific part — do not re-implement query execution. When ≥2 skills need the same deterministic logic, **extract it into the shared script library `learned/hebb_utils/<domain>/`** (a learned artifact, organized by domain, distinct from engine tools in `core/tools/`) and **import** it from each skill's bundled entry script; a composite that ties together logic from several skills' scripts gets a combined module there too. Duplicate a script only as a last resort, when sharing by reference would couple two skills that must stay independent.

   **Decompose by default — don't wait for a second caller.** When a task-specific skill embeds a lower-level capability that is plainly reusable for *other* use-cases (a "read table/log X", "resolve the warehouse routing for model Y", "look up host Z" building block), do **two** things proactively, even when this is its only current consumer: (a) **extract** that capability into `learned/hebb_utils/<domain>/`, and (b) **expose it as its own small skill** so future tasks discover and reuse it — the task-specific skill then becomes a thin caller of the util. The reusable building block, not the count of present callers, is the trigger; a monolith that buries a generic capability inside one task's skill is the anti-pattern to avoid. *Example:* "trace an op to its root" buries a generic "read processor_event_log" capability — extract `hebb_utils/processor/event_log.py` and ship a small `query-processor-event-log` skill alongside the `trace-processor-op` skill, both importing the util.
4. **Extend without breaking.** When you modify an existing skill, changes are **additive** — preserve its current steps, scripts, and contract so existing callers keep working. If a change would alter existing behavior, prefer a new skill over a breaking edit.
   - **Rename / re-describe when an extension outgrows the skill's name or description.** After extending a skill, check whether its `name` and `description` still accurately cover its new scope. If they don't (e.g. an `…-cpu` skill that now also pulls queue-depth), **rename and rewrite the description to fit** — but **keep the existing triggering examples** (they preserve discovery for the original use) and add examples for the new scope. A rename means: rename the skill directory + the `name:` field, update **every reference to the old name** across `learned/` and `core/` (other skills' descriptions/bodies, wiki `## Related skills` and inline mentions — but not the generated `learned/wiki/skills/index.md`, which `publish.py` regenerates), keep the knowledge↔skill symmetry intact, and re-publish so the `.claude/` symlink and the Skills catalog regenerate. A name that no longer matches the capability is itself a defect to fix, not leave.
5. **Write it — author with `skill-creator`, bundle every python script as its own file.** Use `skill-creator` to scaffold and word the `SKILL.md` (structure, steps, a `description` that says *when* to use it so it triggers). Place the result under `learned/skills/<domain>/<name>/`: the `SKILL.md` and, for anything that runs python, a **`scripts/` directory holding the actual `.py` file(s)** — nest them in subfolders however you like. Always add the python **separately** as a committed, bundled artifact — never inline python as a code block in `SKILL.md`, and never leave it as a scratch script.

   **Before authoring a script that calls a vscode function, read the real source — only the specific file the function lives in** (follow the requirement's `proof:` vscode link to it; don't chase imports, don't explore) — to get the signature, imports, and usage right rather than guessing.

   Author every skill **reusable-by-default**: a single, clear, parameterized capability (no session-specific hardcoding) that declares its grounding knowledge in frontmatter as two tiers — `knowledge_required:` (the wiki pages/facts it depends on to function) and `knowledge_optional:` (context that helps if relevant).

   **Pin the model — every learned skill's frontmatter sets `model: sonnet`.** A skill should run on the cheaper model, not the session's premium one; a heavy skill (e.g. `solr-shard-cpu`) otherwise burns a large share of premium tokens every time it runs. Claude Code applies this override only while the skill is active and reverts to the session model afterward, so the saving is free. The field accepts the same values as `/model`; default to `sonnet` and only choose a stronger model if a skill genuinely needs deeper reasoning. When you edit or rename an existing skill, preserve its `model:` line (add it if missing). If part of the capability is a small atomic unit likely reusable by other tasks or systems later, **split it into its own small skill** and have this one call it (keep the small skill independently usable). Bias toward small composable skills over monolithic ones.

   **Importing from `learned/hebb_utils/` without breaking the gate:** the *invoked* entry point stays `${CLAUDE_SKILL_DIR}/scripts/X.py` (so `bash_exec_policy.py` still runs it unattended); it merely *imports* the shared module. Have the entry script **walk up to the dir that contains `hebb_utils/`** (i.e. `learned/`) and put it on `sys.path` before importing (don't hardcode a nesting depth) — e.g. `import os, sys; _d = os.path.dirname(os.path.realpath(__file__))`, then `while not os.path.isdir(os.path.join(_d, "hebb_utils")): _d = os.path.dirname(_d)`, then `sys.path.insert(0, _d)` and `from hebb_utils.<domain> import …`. This keeps the canonical run command (and the gate shape) unchanged.

   **The shared library's import root is `hebb_utils`, not `utils` — so vscode-dependent shared logic works in-process.** vscode ships its own top-level `utils` package (`$CODE_BASE/www/utils`), and two top-level `utils` packages can't coexist on one `sys.path`. Naming the learned library `hebb_utils` keeps it collision-free, so a script can import `hebb_utils.<domain>` **and** vscode code (`from utils.os_constants import …`) in the same process. This matters because **most shared logic is vscode-dependent**: a module that reads `$CODE_BASE` config imports `www` at call time, so its caller runs with `PYTHONPATH=$CODE_BASE/www` (a purely www-free helper can use `PYTHONPATH=$CODE_BASE`; either way `hebb_utils` resolves via the walk-up). Put vscode-dependent shared logic in `learned/hebb_utils/<domain>/` like any other shared logic — **no subprocess needed** (see `solr-shard-cpu`, which imports its host-lookup and CPU stages from `hebb_utils` in one process).
6. **Put only the run command in `SKILL.md`, in the gate-passing shape. Never add user-approval gates.** The body invokes the bundled script as a **single, clean** command that points at the skill's own directory (never hardcode the interpreter):
   ```bash
   PYTHONPATH="$CODE_BASE" "$VSCODE_PYTHON" "${CLAUDE_SKILL_DIR}/scripts/X.py" "$@"
   ```
   This exact shape is what the bash execution policy (`core/tools/bash_exec_policy.py`) recognizes as a **skill-bundled** script and lets run **without a prompt** — so the skill passes the gate every time it is used. Keep it to one command: a chained/compound invocation (`&&`, `;`, `|`, redirects, `$(...)`) is downgraded to a prompt, and any script reached by a non-skill path (a scratch/`/tmp` path, `python -c`) stays gated. Bundling the `.py` under `scripts/` and referencing it via `${CLAUDE_SKILL_DIR}` is exactly what turns a one-time, approval-gated scratch run into a capability that runs unattended next time. `${CLAUDE_SKILL_DIR}` is a **static anchor** to the running skill's own directory, so a reference beneath it passes the gate at **any nesting depth** — the gate keys on the script being anchored under the skill (`${CLAUDE_SKILL_DIR}` or a resolved `skills/` path), not on a flat layout. (It anchors the script's *location*; the run still executes with the shell's working directory, so have scripts resolve their own bundled files relative to `${CLAUDE_SKILL_DIR}` / `__file__`, not the cwd.)

   **Do not add "surface this command for user approval" instructions anywhere in a skill's steps.** Any call a skill needs — including read-only AWS CLI calls (e.g. `aws ec2 describe-instances`, `aws cloudwatch get-metric-statistics`) or other external reads — must be bundled into a `.py` script under `scripts/` so it runs unattended. The only exception is if the user has explicitly asked for an approval gate for a specific operation. If an existing skill has an approval gate that was not explicitly requested, remove it as a capability fix (Rule A4: capability problem → fix the script or steps). When `task-analyser` hands you a `[from-intervention]` gate-removal (a human had to approve a read-only action the agent should have run unattended), removing that unrequested gate **is** the fix.
7. **Promote a scratch script** from the log's evidence into such a bundled script only when its capability is repeatable; keep it deterministic (a script is a pure transform — the skill carries the runtime judgment; Rule A2).
8. **Dynamic-lookup requirements query the live source — never hardcode session-observed values.** When `task-analyser` flags a requirement to fetch live infrastructure state (topology, host-to-shard mapping, instance IDs, service assignments), the bundled script must call the authoritative live API or CLI (e.g. Solr Collections API, `aws ec2 describe-instances`) at runtime. Do not use hostnames, IDs, or assignments observed in the session-doc as constants or defaults — they are ephemeral and will silently go stale.

## Boundaries
- A compile writes **only** under `learned/` (`learned/skills/`, the shared library `learned/hebb_utils/<domain>/`, and `learned/agents/` if A1 forces an agent) — all learned artifacts, so the diff stays traceable to the session-doc; `inputs/` is immutable. You may *reference/import* a core skill or shared tool here. You may **read** `$CODE_BASE` read-only, scoped to the single file a requirement's `proof:` link names, solely to author a script that calls it. The engine itself can be fixed — but that's a separate maintainer task (`core/agents/hebb_injector.md` → *Fixing issues at the source*), not part of compiling this doc into a learned skill.
- One change per requirement, each traceable to the session-doc. Knowledge belongs to `wiki-writer`, not here.

# Hebb — Project Guide

Hebb extends Karpathy's "LLM Wiki" pattern to software-engineering agents. Instead of re-deriving knowledge from the `EightfoldAI/vscode` codebase on every task, Hebb **compiles** the experience of SE agents — recorded as **session-docs** in `inputs/` — once into durable, interlinked artifacts, and keeps them current. Agents **query the compiled artifacts, not the raw history**.

Two goals drive the system, **equally**:
- **Cut human intervention** — every time a human has to step in is a gap to compile away.
- **Cut LLM effort (tokens & time)** — every expensive re-exploration of the codebase is something to turn into a wiki page, a skill, or an automated script so the next task is cheap.

Three maintained artifact types (knowledge is the default; the others are promoted only when warranted):
- **Knowledge → wiki page** (`learned/wiki/`) — "what is true."
- **Capability → skill** (`learned/skills/`) — "how to do X," reusable.
- **Role → agent** (`learned/agents/`) — only when a skill isn't enough; created on demand. Rare.

## The parts of Hebb

| Part | Where | What it is |
|---|---|---|
| **Core engine** | `core/` (`core/skills/`, `core/agents/`, `core/tools/`) | The maintainer's instructions, skills, agents, tools — how the system runs. |
| **Learned artifacts** | `learned/` (`learned/skills/`, `learned/wiki/`, `learned/utils/`) | The compiled **output**, built from witness logs. |
| **Input** | `inputs/` | Witness logs (session-docs) from SE agents. **Immutable** — the raw history the engine compiles from. |
| **Runtime discovery** | `.claude/skills/`, `.claude/agents/` | Symlinks regenerated by `publish.py` — don't hand-edit. |

The **only** immutable path is `inputs/`. Everything else is the maintainer's to edit.

## The two agents — each follows its own manual

| Agent | Role | Operating manual | Invoke |
|---|---|---|---|
| **`hebb`** | SE agent — does the work, writes the witness log | **`core/agents/hebb.md`** | the default agent in a repo session |
| **`hebb_injector`** | Maintainer — compiles a doc into wiki+skills, and fixes the engine | **`core/agents/hebb_injector.md`** | `@hebb_injector inputs/<doc>.md` |

**Read the agent file for your role; its rules live there, not here.** The SE agent and the maintainer operate under *opposite* write rules (the SE agent writes **only** to `inputs/`; the maintainer may edit anything **except** `inputs/`), so the role rules are kept in the agent files to avoid cross-contamination — this guide stays role-neutral. For any maintainer work — compiling a session-doc **or** fixing the engine at the root cause — follow `core/agents/hebb_injector.md` (the full maintainer manual: the injector loop, judgment rules A1–A8, fixing-at-source, the learning loop, the knowledge↔skill graph).

## Command directives

- **Publish** (after any skill/agent change): `python3 core/tools/publish.py` — regenerates the `.claude/` symlinks and the wiki **Skills catalog** (`learned/wiki/skills/index.md`).
- **Lint** (the checker): `python3 core/tools/lint.py` — wiki/skill structural integrity + knowledge↔skill symmetry.
- **Cross-doc report**: `python3 core/tools/intervention_report.py` — interventions per session over time (a prioritization signal).
- **Run a script against the vscode repo** — it's not pip-installed, so put the repo root on `PYTHONPATH`. Canonical invocation (a convention, not shared code):
  ```bash
  PYTHONPATH="$CODE_BASE" "$VSCODE_PYTHON" "${CLAUDE_SKILL_DIR}/scripts/X.py" "$@"
  ```
  `CODE_BASE` (vscode repo root) and `VSCODE_PYTHON` (its venv interpreter) are set in `.claude/settings.json`; `vscode` is an additional working directory. `${CLAUDE_SKILL_DIR}` resolves to the running skill's own directory.
- **Hooks** (`.claude/settings.json`): a `PreToolUse` bash-exec gate (auto-allows vetted skill and `core/tools/` scripts, prompts for scratch), a `SessionStart` hook that injects `learned/wiki/index.md`, and a `Stop` hook that nudges the SE agent to keep its log current.

## Principles
Compile once, maintain forever; query the artifact, not the raw sources. The agent witnesses; the maintainer judges. Optimize **both** axes — fewer human interventions *and* less LLM effort per task — by compiling experience into wiki, skills, and automation. Promote with a high bar (knowledge → skill → agent; script → bundled → shared). When a *compiled* artifact is wrong, fix its **generator**, not just the artifact. Keep it simple — PR review is the only gate.
